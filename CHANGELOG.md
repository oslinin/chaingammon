## [Unreleased]

### Added

- **`docs/team-mode.md`** — design doc for the actual human-in-the-loop feature in Chaingammon: doubles / chouette / human+agent. Per-turn captain rotation, advisor signals (`{teammate_id, proposed_move, confidence, optional_message}`), GameRecord schema extensions, settlement via `MatchEscrow.payoutSplit(matchId, winners[], shares[])` (no team treasury — terms can change every match), and four-phase implementation plan (schema → coach `/chat` extensions → `payoutSplit` Solidity → frontend). Cross-links to `docs/coach-dialogue.md` to disambiguate solo coach (UX) from team mode (training signal).
- **`server/app/game_record.py` team-mode types** — `AdvisorSignal{teammate_id, proposed_move, confidence ∈ [0,1], optional message}` and `Team{members ≥ 1, captain_rotation ∈ {alternating, per_turn_vote, fixed_first}}`. Extensions: `MoveEntry.advisor_signals: Optional[list[AdvisorSignal]] = None`, `GameRecord.team_a: Optional[Team] = None`, `GameRecord.team_b: Optional[Team] = None`. All four new fields are Optional, so `serialize_record(...)`'s `exclude_none=True` keeps the byte-stable invariant — existing solo records on 0G Storage hash to the same Merkle root after the schema change. 17 tests cover field validation, defaults, round-trip through `model_validate_json`, byte-stable serialization (the new fields are omitted when None and emitted when present).
- **`agent/career_features.py` + `sample_trainer.py --career-mode`** — career-mode feature encoder and trainer wiring. `encode_career_context(ctx, dim=16)` projects a `CareerContext` (opponent style, teammate style, stake_wei, tournament position, team-match flag) into a 16-d vector compatible with `BackgammonNet.extras` (slot layout: `[0:6]` opponent style on six axes from `agent_overlay.CATEGORIES`, `[6:12]` teammate style or zeros, `[12]` `log1p(stake)/70` clamped to `[0,1]`, `[13]` tournament position clamped to `[-1,1]`, `[14]` team-match flag, `[15]` bias). `sample_career_context(rng, *, force_team=...)` draws a synthetic context per match (uniform style on `[-1,1]`, log-uniform stake on `[0, 1e21]` wei, 50/50 teammate). The trainer's new `--career-mode` flag swaps the placeholder `encode_extras` random projection for fresh per-match sampled contexts; eval extras stay frozen across the run so `eval/win_rate_vs_frozen` is comparable. 26 encoder/sampling tests + 3 trainer-mode tests (CLI smoke run, `--extras-dim < 16` rejection, encoder consumability by the value net's extras head).
- **`MatchRegistry.settleWithSessionKeysAndSplit`** — trustless session-key settlement WITH on-chain payout via `MatchEscrow.payoutSplit`, closing phase **Team-5** of `docs/team-mode.md`. Same auth message as `settleWithSessionKeys` ("Chaingammon:open" — the session key is authorized at game open before the split is known, so a single human auth signature works for both paths). Result message uses a distinct prefix `"Chaingammon:result-with-split"` so a signature for the non-payout `settleWithSessionKeys` cannot be replayed here, and vice-versa. The new result hash binds `escrowMatchId` and `splitHash = keccak256(abi.encode(winners, shares))` — the relayer cannot tamper with the split arrays without breaking the session key's signature. After auth + result + nonce checks the registry calls `_doRecord` then `_payoutFromEscrow` atomically — escrow revert rolls the whole tx (record + nonce increment + payout) back. 9 new Hardhat tests cover solo single-winner happy path, team-mode N-way split happy path, three tampering revert cases (winners changed, shares changed, escrowMatchId changed), the cross-format replay-protection (a `Chaingammon:result` signature reverts here), nonce mismatch, escrow `ShareSumMismatch` bubbling up with atomic registry rollback, and post-settle nonce replay protection.
- **`MatchRegistry.recordMatchAndSplit` + `setMatchEscrow` wiring** — closes the on-chain settlement loop documented in `docs/team-mode.md`. New `IMatchEscrow` interface (the `payoutSplit` slice MatchRegistry depends on), `matchEscrow` storage, `setMatchEscrow(address)` owner-only setter (idempotent; emits `MatchEscrowSet`), and `recordMatchAndSplit(...recordMatch args, escrowMatchId, winners[], shares[])` owner-only function that calls `_doRecord` then `IMatchEscrow.payoutSplit` atomically — either both succeed or the whole tx reverts (no orphan match records, no orphan escrow state). Single-winner solo and N-way team splits both go through the same path (`payoutSplit` handles N=1 as the degenerate case). New `NoEscrowConfigured` error fires when `recordMatchAndSplit` is called before the wiring. `contracts/script/deploy.js` now calls `registry.setMatchEscrow(escrowAddr)` after deploying both contracts, so a fresh deploy is wired end-to-end. The session-key path (`settleWithSessionKeysAndSplit`) is a follow-up — needs a new signed-message format with the split bound to the resultHash. 9 new Hardhat tests cover setter happy path / non-owner reject / re-wiring, atomic record+payout for solo and team flows, `NoEscrowConfigured` revert, atomic rollback when the escrow call reverts (e.g. match never opened in escrow), non-owner reject, and `ShareSumMismatch` bubbling up from the escrow.
- **`MatchEscrow.payoutSplit(matchId, winners[], shares[])`** — settler-only multi-recipient settlement function for team-mode payouts (see `docs/team-mode.md`). Aligned arrays: `shares[i]` is the wei sent to `winners[i]`. Sum of shares MUST equal the pot (settler pre-computes the split; contract verifies it). Recipients are NOT restricted to the two depositors — team-mode design assumes one address per side stakes (per-match terms can change every match — there is no team treasury) and the team's internal split is decided off-chain, then committed at game-end by `MatchRegistry`. Effects-then-interactions: `settled` flag flipped before any of the N transfers, so reentrant calls hit `AlreadySettled`. Zero-share entries are allowed (settler may want to record team membership without paying) and skipped at transfer time. Zero-address winners revert. New errors: `EmptyWinners`, `LengthMismatch`, `ZeroAddressWinner`, `ShareSumMismatch`. Reuses the existing `PaidOut(matchId, winner, amount)` event (one per non-zero share). 16 new Hardhat tests cover 50/50 split, 3-way custom split summing to pot, non-depositor winners (the team-mode case), zero-share skip, all four new error reverts plus length/match-not-open/already-settled/cross-payout-blocking/no-settler. Lands phase **Team-3** of `docs/team-mode.md`.
- **Team-mode `/chat` kinds** (`agent/coach_dialogue.py`, `coach_service.POST /chat`) — three new `ChatKind` values for the team-mode dialogue flow: `teammate_propose` (the agent acts as advisor and produces a proposed move + confidence + rationale), `teammate_advise` (the agent elaborates on a prior proposal in same-team good-faith framing, distinct from the solo `human_reply` adversarial-defence framing), `captain_decide` (the captain commits a move with optional `chosen_advisor_id` crediting a specific advisor). New `ChatRequest.chosen_advisor_id: Optional[str]` carries the advisor PlayerRef.id when the captain followed someone specifically. Stub replies cover the bar-dance / no-legal-move case for `teammate_propose` and the no-advisor-credited case for `captain_decide`. 8 new dialogue tests + 6 new endpoint tests (45 coach tests total). Lands phase **Team-2** of `docs/team-mode.md`.
- **`agent/coach_dialogue.py` + `coach_service.POST /chat`** — Phase A of the turn-by-turn coach dialogue (see [docs/coach-dialogue.md](docs/coach-dialogue.md)). New data shapes (`DialogueMessage`, `DialogueState`, `ChatRequest`, `ChatResponse`), prompt-assembly helper (`build_chat_prompt`) with per-kind framing for `open_turn` / `human_reply` / `move_committed`, and a session-local keyword-driven preference signal (`update_preferences`) clipped to `[-1, 1]`. Per-session preferences are UX adaptation only; they do not feed agent training (the human-in-the-loop training story is team mode). The `/chat` endpoint runs the full pipeline (prompt assembly → LLM → preference update → delta) with a deterministic stub for the LLM call so the frontend can integrate against the response shape; Phase B replaces the stub with Qwen 2.5 7B on 0G Compute (and flan-t5-base local fallback). 23 dialogue-shape tests + 8 endpoint tests.
- **`docs/coach-dialogue.md`** — four-phase design plan for turn-by-turn coach explanation: per-turn dialogue, agent reads the human, human can talk back, session-local preference adaptation, bounded latency. Documents data shapes, conversation flow, where each piece lives, cost/latency budget, explicit non-goals. Cross-links to `docs/team-mode.md` for the actual human-in-the-loop story.
- **`agent/rules_engine.py`** — pure-Python backgammon move legality. The validator KeeperHub's `validate_move` step relies on; the WASM bundle in the browser and the KeeperHub validator's WASM module both compile from the same logic, so all three observers (browser, KeeperHub, audit replayer) agree bit-for-bit. Public API: `Board`, `parse_move`, `dice_pool`, `is_legal`. 28 tests cover parse shapes, dice pool, board predicates, legal/illegal happy + edge cases (bar entry, hits, bear-off, doubles, robustness).
- **`docs/keeperhub-workflow.md`** — canonical workflow spec.
- **`docs/keeperhub-workflow.schema.json`** — machine-checkable JSON Schema for `kh run status --json` payloads. Codifies the 14 step-id enum, status enum, retry-count constraints, hash patterns. Validated by `server/tests/test_keeperhub_workflow_schema.py` (10 cases including representative running + settled workflows and 5 negative cases). `jsonschema` added as a dev dep on `server/pyproject.toml`.
- **`server/app/game_record.py` extensions** — `MoveEntry.drand_round` (Optional int) per move so an audit replayer can re-derive dice via `agent.drand_dice.derive_dice`; `GameRecord.series` (Optional `SeriesEnvelope`) so a match can carry `{series_id, series_index, series_total}` for best-of-N or tournament-round matches. Backwards-compatible because `serialize_record` uses `exclude_none=True` — existing records on 0G Storage hash to the same Merkle root after the schema change.
- **`agent/sample_trainer.py`** — runnable end-to-end demo of the per-agent value-network training loop. Defines `BackgammonNet` (gnubg-init core + per-agent random extras head + scalar equity head); instantiates two networks that share the same gnubg-initialised core but have different random extras (the "same gnubg base, different personality" picture); runs self-play TD(λ) with eligibility traces; logs scalars (TD error, value estimates, eligibility-trace norm, gradient norm, win-rate vs the frozen opponent), parameter and gradient histograms, and the model graph to TensorBoard via `torch.utils.tensorboard.SummaryWriter`. CLI flags: `--launch-tensorboard`, `--save-checkpoint` / `--load-checkpoint`, `--drand-digest`, `--upload-to-0g`. Includes a tiny in-file `RaceEnv` so the demo runs anywhere without a backgammon engine.
- **`agent/drand_dice.py`** — pure-Python helper that derives backgammon dice from a drand round digest: `dice = keccak256(round_digest ‖ turn_index_be8) mod 36`. Wired into `sample_trainer` via `--drand-digest`; production code consumes it through KeeperHub-pulled rounds.
- **`agent/checkpoint_encryption.py`** — AES-256-GCM wrap/unwrap for trainer checkpoint blobs. Format: `nonce(12) ‖ ciphertext_with_tag`. The encrypt/decrypt step in the README's "AES-GCM encrypt weights → upload to 0G Storage" lifecycle. `cryptography>=47.0.0` added to `agent/pyproject.toml`.
- **`agent/og_storage_upload.py`** — agent-side helper that shells out to the `og-bridge` Node CLI to publish a (typically encrypted) blob to 0G Storage and return `UploadResult(root_hash, tx_hash)`. Mirrors the subprocess pattern in `server/app/og_storage_client.py` and `agent/coach_compute_client.py`.
- **`agent/og_storage_download.py`** — symmetric of upload. Fetches a blob by Merkle root via `og-bridge/src/download.mjs`. Closes the resume-training read path (sample_trainer's `--init-from-0g` consumes it).
- **`agent/sample_trainer.py --init-from-0g`** — resume training from a 0G Storage checkpoint: fetch the sealed blob, decrypt with `--init-key`, write to a temp file, and `load_checkpoint`. `match_count` is preserved so progress accumulates across resume cycles.
- **`docs/keeperhub-workflow.md`** — canonical KeeperHub workflow spec: per-step inputs/outputs/gas/retry policy for the `match_open` → `per_turn_loop` → `match_settle` graph. Documents drand pull, WASM rules-engine validation, `settleWithSessionKeys`, `payoutWinner`, ENS text-record updates, audit-trail writes; failure modes; cross-links to the reference implementations.
- **`MatchEscrow` deploy + verify wiring** — `contracts/script/deploy.js` now deploys `MatchEscrow` with `settler = MatchRegistry`, and `verify.js` recovers the constructor args from the deployment JSON. The actual `MatchRegistry → MatchEscrow.payoutWinner` hook is a follow-up.
- **`agent/tests/test_sample_trainer.py`** (12 cases), **`agent/tests/test_drand_dice.py`** (10), **`agent/tests/test_checkpoint_encryption.py`** (14), **`agent/tests/test_og_storage_upload.py`** (9) — 45 new agent tests covering the trainer + helpers.
- **`scripts/fetch_drand_round.py`** — manual / debugging companion to the trainer's `--drand-digest` flag. Pulls a public drand round (League of Entropy mainnet) and prints its digest in hex.
- **`contracts/src/MatchEscrow.sol`** + **`contracts/test/phase_MatchEscrow.test.js`** — per-match stake escrow with `deposit` / `refund` / `payoutWinner`. Matches keyed by `bytes32 matchId` (off-chain `keccak256(playerA ‖ playerB ‖ nonce)`). Settler-only payout; refunds allowed only before the match opens. 15 hardhat tests cover deposit / refund / payout happy paths and revert cases.
- **`tensorboard>=2.16.0`** dependency added to `agent/pyproject.toml` for `SummaryWriter` + the dashboard binary.
- **`runs/`, `*.ckpt`, `*.pt`** added to `.gitignore` — TensorBoard event files and trainer checkpoints are per-run, large, and should not be tracked.

### Fixed

- **Sepolia default RPC in `contracts/hardhat.config.js`** — was `https://rpc.sepolia.org` (Cloudflare 522 by late 2025); now `https://ethereum-sepolia.publicnode.com`. Documents `SEPOLIA_RPC_URL` in `contracts/.env.example`.

### Changed

- **README.md** — concise rewrite (863 → 408 lines) aligned with `chaingammon_plan.md`. Replaces the obsolete "shared frozen gnubg + per-agent overlay" framing with the per-agent trained-NN model: gnubg weights as starter init, TD(λ) self-play training, browser inference by default with 0G Compute (TEE-attested) for offline play. Re-targets settlement to Sepolia (KeeperHub-native) and adds drand as the dice VRF (each turn's roll is `keccak256(drand_round_digest, turn_index) mod 36`). Splits the protocol-roles table into sponsor protocols (0G, ENS, KeeperHub — the three Chaingammon targets at ETHGlobal Open Agents) and other infrastructure (Sepolia, drand). Drops the redundant Mission/Motivation/Advantages sections.
- **CONTEXT.md architecture diagram** mirrors the README's target architecture.
- **Removed all AXL / Gensyn AXL references** from code and documentation. AXL was described as the relay layer between the browser and the local agent services in earlier drafts but was never actually used (the browser hits `localhost:8001` / `:8002` directly). Deleted `agent/axl-config.json` and four stale plan/spec files under `docs/superpowers/`. Rewrote the `start.sh` startup script and the docstrings in `agent/{gnubg_service,coach_service,gnubg_state}.py`.

- **Phase 20 — 0G Compute coach with agent-bias awareness.** Replaces the local-only flan-t5-base coach with verifiable inference on 0G Compute (Qwen 2.5 7B Instruct) via a new `og-compute-bridge` Node CLI. Adds an `AgentProfile` abstraction (`agent/agent_profile.py`) that pulls each agent's experience overlay from 0G Storage and renders its top biases as a one-sentence prompt context — forward-compatible with the `learn` branch's PyTorch model. Coach panel now has a Paid · 0G / Free · Local toggle (persisted in `localStorage`); the server falls back from compute → local invisibly when 0G is unreachable, and surfaces the served backend in the hint card.
- **Phase 18 — match page over local gnubg agent process.** Browser holds match state; routes every move through `gnubg_service` (`/new`, `/apply`, `/move`, `/resign`) on `localhost:8001`; rolls dice client-side via `crypto.getRandomValues`. Removes the last dependency on the retired FastAPI server. (`frontend/app/match/page.tsx`, `frontend/app/dice.ts`)
- **Phase 17 — local agent processes** (`agent/gnubg_service.py`, `agent/coach_service.py`). gnubg service exposes `/move` / `/evaluate` plus the new `/new` / `/apply` / `/resign` endpoints over a unified `MatchState` shape. Coach service runs flan-t5-base for one-to-two-sentence move narrations. Python project is `uv`-managed.
- **Network dropdown in navbar.** Lets the user switch between 0G Galileo Testnet, Sepolia, and (in dev) Hardhat Localhost from the connect button area. Replaces the one-shot "Switch to X" amber nudge.
- **`/test-network-dropdown` fixture page** + Playwright spec covering dropdown variants.
- **`home-navbar.spec.ts`** — positive Playwright assertion that the home page renders one of the three connect-state UIs (catches a class of regressions where `ConnectButton` silently fails to render).
- **`allowedDevOrigins`** in `next.config.{js,ts}` so the dev server accepts cross-origin requests from LAN hostnames.
- **`scripts/upload_gnubg_docs.py`** — one-time uploader for the gnubg strategy doc consumed by `coach_service` as RAG context.

### Changed

- **Frontend match page** rewritten end-to-end for the post-pivot local-agent flow. New `MatchState` interface (drops `game_id`, `cube`, `cube_owner`); calls `NEXT_PUBLIC_GNUBG_URL` (default `http://localhost:8001`) instead of the retired FastAPI server.
- **`MatchRegistry.recordMatch`** records `winnerHuman` / `loserHuman` addresses with zero-address slots when an agent plays — same shape, kept compatible with the upcoming two-sig design.
- **Pivot (`7053a9fb`):** dropped the central FastAPI server and KeeperHub workflow in favor of local agent processes (gnubg + coach FastAPI services on `localhost`) and a two-sig (or future state-channel) on-chain settlement.

### Fixed

- **Sepolia default RPC.** Replaced dead `https://rpc.sepolia.org` (Cloudflare 522 / connection timeout) with `https://ethereum-sepolia.publicnode.com`. The `NEXT_PUBLIC_SEPOLIA_RPC_URL` env override is unchanged.
- **SSR/client hydration mismatch in `ConnectButton`.** Defers real render until after client `useEffect` flips a `mounted` flag, so the SSR (no `window.ethereum`) and post-hydration trees agree. Without this, React silently dropped the click handler.
- **Cross-origin LAN dev access.** Next 16 blocks `_next/*` resources from non-localhost origins by default; `allowedDevOrigins` in `next.config.*` whitelists LAN hostnames so JS bundles + HMR load correctly when accessing from another device.
- **`upload_gnubg_docs.py`** loads `server/.env` via `python-dotenv` automatically. Eliminates the "Missing env vars for 0G Storage" failure when running the script directly.

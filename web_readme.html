Phase 1

gnubg Wrapper Service

FastAPI drives gnubg via pexpect. Full match playable through REST endpoints.

Done ✓

The game server is the load-bearing piece of the whole system — every piece of on-chain logic depends on gnubg producing legal positions, valid move strings, and correct dice. This phase wraps the gnubg binary via pexpect (a Python subprocess expect library), exposes it as a FastAPI service, and defines the canonical GameState model that all subsequent phases consume. The gnubg binary is a robust, heavily-tested open-source backgammon engine; by wrapping it in a subprocess rather than reimplementing it, Phase 1 offloads all position validation and equity calculation to a mature implementation.

gnubg's native move notation (8/5 6/5, bar/N, N/off) is passed through unmodified to the REST API and later to the frontend. This transparency makes test reproduction easy — any failing position can be replayed by hand with gnubg's interactive CLI.

Phase 2

Core Contracts

EloMath library · MatchRegistry · AgentRegistry (ERC-721 stub)

Done ✓

Defines the on-chain data model: ELO math as a pure Solidity library, a match registry that stores results and updates ratings, and an ERC-721 agent registry. EloMath uses integer arithmetic with a lookup table for expected scores, avoiding floating-point which the EVM doesn't support. Every future phase that touches ratings or agent identity builds on these contracts — no later phase changes the EloMath algorithm or the core MatchInfo struct layout. The contracts compile and test against Hardhat's local in-process node at this stage; Phase 4 deploys them to a live testnet.

Nothing is live on 0G testnet yet in Phase 2. The deploy script runs against in-process Hardhat only. Persistent live addresses land in Phase 4, and the Phase 5 redeploy supersedes those.

Phase 3

Game-Record Hash on MatchRegistry

MatchInfo gains a bytes32 gameRecordHash slot. Enables cryptographic tie between on-chain match and 0G Storage archive.

Done ✓

Adds a bytes32 slot to MatchRegistry to hold a 0G Storage Merkle root hash alongside each match record. No upload logic yet — that arrives in Phase 6. The design decision to add the field first (Phase 3) rather than alongside the upload (Phase 7) means the contract's storage layout is stable from this point: any tooling that reads MatchInfo doesn't need to be updated when upload logic is added later. The new GameRecordStored event allows off-chain indexers to subscribe separately from ELO-update events.

bytes32(0) is explicitly permitted so the contract remains usable before Phase 6 wires up uploads, and the 32 prior tests pass unchanged by simply passing ZeroHash as the new argument.

Phase 5

AgentRegistry as ERC-7857 iNFT

Each agent token carries two data hashes pointing at encrypted intelligence on 0G Storage. Divergent overlays = divergent playing styles.

0G Storage 0G Chain
Done ✓

Upgrades AgentRegistry from a plain ERC-721 to the ERC-7857 iNFT shape — each token carries two cryptographic pointers: dataHashes[0] to the shared gnubg base weights (Phase 8 uploads the real file), and dataHashes[1] to this agent's growing experience overlay (Phase 9 populates it after each match). The tier field (0=beginner … 3=world-class, immutable at mint) controls gnubg search depth. This is the contract foundation for the core user-facing claim: two agents minted at the same tier will diverge into measurably different playing styles as their match histories accumulate. The full ERC-7857 transfer-with-reencryption-proof flow is out of scope for v1; only the data-hash shape is implemented.

Phase 6

0G Storage Bridge

og-bridge Node.js package wraps @0gfoundation/0g-ts-sdk. Python server shells out via subprocess.

0G Storage
Done ✓

0G Storage is the decentralized storage layer of the 0G network — clients upload bytes and get back a Merkle rootHash that anyone can use to fetch those bytes, with no login or API key. The only officially supported SDKs are TypeScript and Go; there is no native Python SDK. Rather than rewriting the SDK, this phase introduces og-bridge: a tiny Node.js workspace package that exposes two stdin/stdout CLI scripts backed by the official @0gfoundation/0g-ts-sdk. The Python server shells out to these scripts via subprocess, isolating the language boundary to a clean pipe interface. SDK progress logs are redirected to stderr so stdout stays parseable. This pattern is reused by all subsequent 0G Storage uploads (weights in Phase 8, overlays in Phase 9).

The round-trip test (~14 s) runs against 0G Storage testnet. Most of that time is waiting for the on-chain transaction that pins data to 0G Storage's flow contract. The test is automatically skipped when OG_STORAGE_PRIVATE_KEY is unset, so CI without secrets stays green.

Phase 7

Archive & Finalize

POST /finalize builds GameRecord → uploads to 0G Storage → calls MatchRegistry.recordMatch with rootHash. On-chain match is now cryptographically tied to off-chain replay.

0G Storage 0G Chain
Done ✓

The keystone of the on-chain integrity claim: after a match ends, the server serializes the full play sequence (moves, dice, positions) into a canonical JSON envelope, uploads it to 0G Storage to get a deterministic Merkle rootHash, then calls MatchRegistry.recordMatch with that hash embedded. Anyone with the match ID can read the hash on-chain and re-download the original game record from 0G Storage — no login, no API key, no platform. The serialization uses sorted keys and UTF-8 encoding so the same record always produces the same Merkle root; the rootHash is the content-addressed identity of the game. ChainClient is a thin web3.py wrapper that signs recordMatch directly as the deployer wallet in v1 — Phase 17/18 will route this through KeeperHub.

Phase 8

Encrypted gnubg Weights

AES-256-GCM envelope uploaded to 0G Storage. Hash pinned on AgentRegistry via setBaseWeightsHash. Every agent's dataHashes[0] is now cryptographically verifiable.

0G Storage 0G Chain
Done ✓

The gnubg neural-network weights file (/usr/lib/gnubg/gnubg.wd, ~399 KB) is the shared intelligence base every agent token runs against. This phase encrypts it with AES-256-GCM, uploads the ciphertext envelope to 0G Storage, and pins the resulting rootHash on AgentRegistry via setBaseWeightsHash. Every agent's dataHashes[0] now points at the same verifiable blob — anyone can read the hash on-chain, download the blob from 0G Storage, decrypt with the known key, and confirm they get the same gnubg weights. The version byte (0x01) in the envelope is reserved so a v2 upgrade to per-owner hybrid encryption can change the layout without breaking v1 readers.

The encryption key is server-held in v1 — anyone with the key can decrypt the blob from 0G Storage. This is intentional because every agent runs the same shared base; there's no per-owner secret to protect. v2 would use hybrid encryption (per-iNFT-owner capability) once agents carry unique weights.

Phase 9

Experience Overlay — iNFTs That Learn

Each agent iNFT carries a preference vector that biases its move choices. After every match the overlay is updated, uploaded to 0G Storage, and committed on-chain. Two agents on the same base weights diverge into measurably different styles.

0G Storage 0G Chain
Done ✓

Makes each agent iNFT a living asset: after every match, a damped-reinforcement update biases the agent's move choices in the direction of moves that correlated with wins. The overlay is a 20-category preference vector (values in [-1,1]) stored as deterministic canonical JSON on 0G Storage; the rootHash is committed on-chain via updateOverlayHash after each match. With a zero overlay, an agent plays vanilla gnubg. As wins accumulate, the overlay tilts the agent toward its winning style. Every match is an immutable experienceVersion entry — the iNFT's on-chain history of how it learned. The overlay biasing happens entirely outside gnubg: gnubg produces a ranked candidate list, apply_overlay re-scores it using the preference vector, and the top-scored move is submitted.

The overlay is a tendency tracker, not a neural network or RL policy. It tracks which ~20 behavioral categories (point-building, running, hitting, etc.) correlate with this agent's wins. The category list is frozen in v1 — adding categories at the end is safe (old blobs round-trip with new entries zero-filled), but reordering or removing would invalidate all existing overlays.

Update Rule

Damped reinforcement — early matches move overlay a lot; late matches barely shift it. alpha = N / (N + match_count). Clips to [-1, 1]. Exposure-normalized so a 50-move match ≠ 50× more update than a 5-move match.

Phase 10

ENS Subname Registrar

PlayerSubnameRegistrar — issues <label>.chaingammon.eth subnames with ENS-shaped text records. Deployed on 0G testnet; v2 can mirror to real ENS on Sepolia/Linea without rewriting.

ENS 0G Chain
Done ✓

Portable reputation requires portable identity. ENS subnames (e.g. alice.chaingammon.eth) are the standard for this in the EVM ecosystem — their text records can hold arbitrary key-value data that any ENS-aware tool can read. PlayerSubnameRegistrar issues subnames under chaingammon.eth on 0G testnet, with ENS-compatible namehash computation (keccak256(parentNode || keccak256(label))) and dual-auth text record writes: the contract owner (the server) can push ELO after each match, while the subname holder can update their own profile. The contract is deliberately self-contained on 0G testnet — a v2 deployment could mirror to real ENS on Sepolia via a Durin-style L2 registrar without rewriting any contract logic.

We can't realistically own chaingammon.eth on a chain with a live ENS root within the hackathon timeline. The contract's interface is ENS-shaped (namehash, text records, resolver semantics) so the portable-identity story works as a demo and the upgrade path to real ENS is clear. Live deploy of this contract is deferred one phase (deployed in Phase 11's deploy_registrar.js) to avoid redeploying AgentRegistry and wiping agent #1's overlay state.

Gensyn AXL ENS 0G Chain 0G Storage
Done ✓

The FastAPI server and KeeperHub are removed entirely. gnubg and an flan-t5-base LLM coach now run as AXL agent nodes — FastAPI services exposed over the Gensyn AXL (Agent eXchange Layer) P2P mesh. AXL is a single binary built on Yggdrasil that gives each node an end-to-end encrypted identity; the browser talks to localhost:9002 and AXL routes requests to whichever gnubg/coach node is available. Human-vs-human game relay also uses AXL A2A — no WebRTC, no 0G KV signaling. Settlement moves to a two-signature on-chain call: both players sign the result off-chain, either submits both sigs to MatchRegistry.recordMatch, the contract verifies via ecrecover and updates ELO atomically. No orchestrator, no single point of failure.

This commit contains only the spec and plan documents. Implementation (Tasks 1–9) follows in subsequent commits.

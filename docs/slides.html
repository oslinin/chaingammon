The Problem

Your backgammon rating is not yours. Spend years climbing the ladder on any platform and that rating lives in their database — locked behind their login wall, gone if they shut down. Switch platforms and you start at zero.
  • Siloed ELO. backgammon.com, PlayOK, and every online club run separate databases with no portability.
  • Opaque AI opponents. No verifiable proof of an AI's intelligence tier, history, or learning.
  • Trusted operators everywhere. Server rolls dice. Server records results. Server can cheat or disappear.
  • No composable reputation. Other tools can't read your rating without a platform-specific API key.

The Solution — an open protocol

Every player — human or AI agent — has an ENS subname (<name>.chaingammon.eth) whose text records hold their ELO and a link to their full match archive on 0G Storage. Any tool can read it without permission.

Verifiable

Every match settles to MatchRegistry. ELO delta + game record hash are on-chain and auditable by anyone.

Portable

ENS subname = identity. Switch frontends, switch clients — reputation travels with your wallet.

Decentralised

No central game server. Local AXL nodes handle move evaluation and coaching. Settlement needs no operator key.

Original plan: FastAPI server rolls dice, records matches, runs gnubg. Two design pivots turned every layer trustless.

Pivot 1 — Gensyn AXL removes the central server

Each player runs two local AXL agent nodes. The browser talks to them over Gensyn's permissionless Yggdrasil P2P mesh.

gnubg agent · port 8001

Move evaluation

Wraps gnubg via its External Player interface. POST /move → best legal move. POST /apply → validated state transition. POST /evaluate → ranked candidates for the coach.

coach agent · port 8002

LLM narration

flan-t5-base + gnubg strategy docs on 0G Storage as RAG context. POST /hint → plain-English coaching hint rendered in the match UI after every move.

AXL routes browser HTTP to whichever service is registered under those names. The player can audit the code. No operator is in the critical path for move evaluation or coaching.

Pivot 2 — settleWithSessionKeys

Trustless settlement without MetaMask interruptions during play and without a DoS vector.

  • Game start — one MetaMask popup. Wallet signs "Chaingammon:open" + nonce + agentId + sessionKey. The ephemeral session key is generated in-browser and lives only in JS memory for this match.
  • During play — no popups. The session key signs per-turn state updates silently.
  • Game end — one MetaMask popup. Browser calls settleWithSessionKeys(humanAuthSig, resultSig, agentId, ...).
The contract recovers the human wallet address from humanAuthSig, verifies that resultSig came from the declared sessionKey, and consumes the nonce. The loser cannot DoS by going offline — they pre-signed at game start, and either side (or any relayer) can submit the result.

AI Agents as ERC-7857 iNFTs on 0G Chain

Each agent is the token — not a label pointing at an off-chain model. Transfer the token, transfer the brain.

dataHashes[0]

Shared gnubg weights

The same encrypted gnubg neural-net file (≈ 400 KB) lives on 0G Storage as a shared blob. All agents reference the same hash. Verifiable: anyone can decrypt and confirm it's the real gnubg weights.

dataHashes[1]

Per-agent experience overlay

A ~50-float preference vector (slot tendency, cube aggressiveness, bear-off timing, …). Starts at zero. Rewritten after every match via damped-reinforcement update. Two same-tier agents diverge in measurable style after 50 matches.

The tier field (0 = beginner … 3 = world-class) maps to gnubg search depth. Tier + overlay hash = the agent's unique intelligence fingerprint, permanently committed on-chain.

ENS — Portable Player Identity

Any third-party tool that knows a player's ENS name can reconstruct their full reputation from public on-chain data — no API key, no Chaingammon account, no platform lock-in.

0G Storage — the Match Archive

Every completed match is preserved as a canonical, content-addressed entry on 0G Storage. The on-chain MatchRegistry carries only the Merkle root hash as a cryptographic pointer.

Per-match Log entry

GameRecord envelope: every move, dice roll, position ID, timestamp. JSON, sorted keys, deterministic bytes. ~2–10 KB compressed per match.

Per-agent Blob

Encrypted gnubg weights (shared base, uploaded once) and the per-agent experience overlay. Hash committed to the iNFT's dataHashes array.

Coach RAG context

gnubg strategy documentation uploaded once by scripts/upload_gnubg_docs.py. The coach fetches it by hash as prompt context.

Per-player KV

Style profile blob: opening tendencies, cube aggressiveness, bear-off speed. Read by the coach for opponent-aware advice.

The LLM Coach

agent/coach_service.py — FastAPI on port 8002, served through AXL. After every move the frontend requests a coaching hint.

How it works: browser calls POST /evaluate on gnubg_service to get ranked candidates, then POST /hint on coach_service with those candidates. flan-t5-base generates a 1–2 sentence plain-English explanation of the best move, grounded in the gnubg strategy docs blob from 0G Storage.
  • RAG context. Strategy doc blob uploaded once to 0G Storage; fetched by its hash for every hint. Keeps the LLM grounded in real backgammon theory.
  • Best-effort delivery. The coach panel shows "Thinking…" while the hint is loading and "Start the coach node" when the service is offline. The game continues regardless.
  • No hosted inference key. flan-t5-base (~250 MB) runs locally on the player's machine. First run downloads and caches it via HuggingFace.

Advantages

For players

  • Own your rating — ENS subname, not a platform row
  • No central server to trust or lose
  • One MetaMask popup per match (session key)
  • Portable across any Chaingammon-compatible frontend

For agent owners

  • Token = intelligence, not a label
  • Agents learn match by match via experience overlay
  • Verifiable provenance: anyone can confirm gnubg weights
  • Transfer token → transfer brain + history

For the ecosystem

  • Open protocol — build a frontend in a weekend
  • ELO as a composable primitive (DNS for skill)
  • Public game archive on 0G Storage — no login required

Security

  • DoS-resistant settlement: loser can't refuse
  • Per-address nonce blocks settlement replay
  • Chain ID + contract address in signed messages blocks cross-chain replay

Limitations (v1)

Trust assumptions still present

  • Dice rolled in browser — honest but not provably fair (commit-reveal = v2)
  • onlyOwner recordMatch co-exists with settleWithSessionKeys as a fallback
  • Session key in browser memory — page refresh loses it

Scope not yet implemented

  • Human-vs-human requires AXL peer discovery (mesh exists; matchmaking doesn't)
  • Coach uses flan-t5-base locally; 0G Compute (verifiable inference) is roadmap
  • ENS on 0G testnet, not wired into real ENS on Sepolia
  • No cube doubling UI
  • Overlay categories are hand-coded heuristics, not learned features
